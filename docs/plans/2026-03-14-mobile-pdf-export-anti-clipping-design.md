# Mobile PDF Export Anti-Clipping Design

## 背景

手机端 PDF 目前已经改成捕获隐藏的 A4 导出容器，页数显著优于直接截取手机窄屏预览。但最新反馈表明，导出的文字与行内容会被局部裁切，典型表现是：

- “标准答案”所在行上下被裁掉一部分。
- “根因”“错误定位”等单行文本底部被截断。
- 现象出现在手机 PDF 中，屏幕预览与桌面打印并无此问题。

## 问题判断

这次问题更像是隐藏导出容器在 `html2canvas` 栅格化阶段出现的排版裁切，而不是分页切线问题。理由是：

- 被裁掉的是单行内部的文字高度，而不是题块整体在页边界被切。
- 截断发生在 `inline-field`、`MarkdownRenderer`、紧凑行高和导出容器样式的组合位置。
- 同一套内容在屏幕预览和桌面打印中可正常显示，说明核心数据和正常 DOM 布局并未损坏。

## 目标

- 只调整移动端隐藏导出容器的排版，避免单行文字和公式被裁切。
- 保持屏幕预览不变。
- 保持桌面端 `window.print()` 打印布局不变。
- 允许手机 PDF 比当前版本略松一点，但不能回退到最初的超长 6 页版本。

## 方案对比

### 方案 A：仅放松导出容器样式，推荐

- 提高导出容器专用 `line-height`。
- 让 `inline-field` 在导出容器里使用更安全的块/行内块布局。
- 放宽导出容器里 `MarkdownRenderer` 的段落与行内容设置，减少 `display: inline` 和过紧的 `leading` 组合。

优点：
- 风险最小。
- 不影响正常页面和桌面打印。
- 最符合当前症状。

缺点：
- 会让导出 PDF 稍微松一点。

### 方案 B：继续深挖分页算法

- 继续围绕 `buildPdfPageSlicesFromBlocks(...)` 调整。

优点：
- 能继续优化“整题不切断”。

缺点：
- 对当前“单行被裁”问题帮助有限。
- 容易继续在错误方向上消耗时间。

### 方案 C：彻底改掉 html2canvas

- 换成别的 PDF 生成方式。

优点：
- 理论上可彻底控制输出。

缺点：
- 工程量太大，不适合当前问题。

## 决策

采用方案 A。

## 设计细节

### 1. 导出容器专用行高

- `.pdf-export-sheet` 的 `line-height` 从当前紧凑值提高到更安全的区间。
- 针对 `.inline-field`、`.inline-field-content`、`.markdown-content p` 再单独加一点垂直空间。

### 2. 导出容器专用内联字段布局

- 导出容器里的 `inline-field` 改成更稳健的对齐方式，避免 `flex + inline paragraph + 紧凑 line-height` 造成裁切。
- 优先使用 `inline-block` / `block` 组合而不是把段落强行压成 `display: inline`。

### 3. 导出容器里的 Markdown 放松

- 对导出容器里涉及 `标准答案`、`错误定位`、`根因` 的 Markdown 输出，避免使用过于紧凑的行高。
- 必要时给这些字段增加最小顶部/底部 padding。

### 4. 范围控制

- 本次不修改屏幕预览样式。
- 不修改桌面 `@media print` 样式。
- 保留现有移动端 A4 导出容器和现有分页逻辑，先只解决“裁字”问题。

## 验证

- 复测此前被裁掉的题目：
  - “标准答案”整行应完整显示。
  - “错误定位”“根因”不应再出现上下截断。
- 页数允许略有增加，但不应回退到最初过长版本。
