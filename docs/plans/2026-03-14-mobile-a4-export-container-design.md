# Mobile A4 Export Container Design

## 背景

当前手机端的 PDF 导出是直接截图当前可见的打印预览页面。桌面端依赖 `window.print()`，会触发 `print` 媒体查询与 A4 排版；手机端则捕获窄屏单列布局，所以同一批题目在手机 PDF 里会被拉成长页面，页数明显多于桌面端。

## 问题定义

- 手机端导出的 PDF 不是 A4 / 桌面布局，而是当前手机布局的截图。
- 题目在手机窄屏下堆成单列，导致每题高度显著增加。
- 现有 `printContentRef` 同时承担“可见预览”和“移动端截图源”两个职责，无法分别优化屏幕显示与导出排版。

## 目标

- 桌面端保持现有 `window.print()` 行为不变。
- 手机端 PDF 改为捕获一个隐藏的、固定桌面/A4 宽度的导出容器。
- 导出的内容尽量贴近桌面打印效果，包括两栏布局、字号、间距和每题密度。

## 方案对比

### 方案 A：隐藏 A4 导出容器，推荐

- 页面继续保留当前可见预览，供桌面与手机浏览。
- 额外渲染一个只用于导出的隐藏容器，强制使用桌面/A4 布局。
- 手机端点击导出时，`html2canvas` 捕获隐藏容器，而不是捕获当前可见页面。

优点：
- 不影响桌面打印链路。
- 手机导出可以稳定复用桌面排版。
- 后续若继续优化 PDF 密度，只需调导出容器样式。

缺点：
- 需要把打印页内容抽成共享渲染，避免维护两份 JSX。

### 方案 B：在手机上临时切换当前页面到桌面布局再截图

- 导出前动态给当前页面打上“桌面模式” class。
- 截图完成后再恢复手机布局。

优点：
- 代码改动表面上较少。

缺点：
- 导出时页面会闪动。
- 容易和响应式 state、滚动位置、图片加载时机互相干扰。
- 失败恢复路径更复杂。

### 方案 C：真正服务端/前端重新排版生成 PDF

- 不再截图 DOM，而是单独生成 PDF 排版。

优点：
- 输出最可控。

缺点：
- 工程量明显偏大，超出当前问题范围。

## 决策

采用方案 A。

## 设计细节

### 1. 共享打印内容渲染

- 把每道题的渲染抽成共享组件或内部渲染函数。
- 屏幕预览与导出容器共用同一份数据与结构，只通过 `layout` 参数切换 class。

### 2. 隐藏导出容器

- 导出容器不使用 `display: none`，避免 `html2canvas` 无法捕获。
- 使用脱离视口的固定定位，例如 `fixed left-[-10000px] top-0`。
- 显式设置接近 A4 的桌面宽度，例如 `1120px` 左右，并强制两栏布局。
- 导出容器仅在移动端导出时被引用，平时对交互无影响。

### 3. 布局策略

- `review` 模式导出容器固定走 `50% / 50%` 两栏。
- `redo` 模式导出容器固定走 `40% / 60%`。
- 导出容器继承 `.print-sheet` 的紧凑字体与段落规则，不依赖真正的 `@media print` 才生效。

### 4. PDF 捕获策略

- 桌面端：继续 `window.print()`。
- 手机端：`html2canvas` 只截隐藏 A4 导出容器。
- 继续保留安全主题变量覆盖，避免 `lab()` 等颜色函数兼容问题。

### 5. 风险与边界

- 隐藏容器与可见容器双渲染会增加一点 DOM 体积，但打印页数据量本来有限，可接受。
- 图片较多时需要确保导出容器图片已加载完成；首次实现先复用现有行为，若仍有截取竞态，再补图片加载等待。
- 本次不改桌面打印分页逻辑，也不引入新的 PDF 布局引擎。

## 验证

- 在手机端导出同一批题目，页数应明显接近桌面端。
- 桌面端按钮仍然弹出浏览器打印对话框。
- 手机端导出的内容不应再出现窄屏单列拉长问题。
