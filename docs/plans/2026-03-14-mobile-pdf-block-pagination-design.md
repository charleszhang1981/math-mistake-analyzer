# Mobile PDF Block Pagination Design

## 背景

手机端 PDF 已经改成捕获隐藏的 A4 桌面布局容器，页数显著缩短，但当前分页仍然沿用“整张长图按固定像素高度切片”的策略。结果是只要切片线落在题块中间，就会把同一道题在上下两页之间硬切开，造成答案、错误定位、根因等内容被截掉。

## 问题定义

- 当前 `buildPdfPageSlices(...)` 只知道画布总高度和 A4 页高，不知道“题目边界”。
- 打印页的每道题实际上是天然的分页块，但导出算法没有利用这层结构。
- 用户不希望普通题被切成两半；允许适度页尾留白，但不接受内容被截断。

## 目标

- 普通题在手机 PDF 中按整题分页，不在题目中间硬切。
- 只有当“单题本身就超过一页”时，才退回到题内切片。
- 桌面端打印链路保持不变。

## 方案对比

### 方案 A：按题块分页，推荐

- 在导出容器中给每道题打上可测量的块标记。
- 导出时读取每题在导出容器中的 `offsetTop` 与 `height`。
- 先按题块边界计算分页；若某题放不下当前页但单题可放下一整页，则整题移到下一页。
- 只有单题高度超过单页高度时，才在这题内部继续硬切。

优点：
- 大多数题不会被切断。
- 余白可控，且只出现在题块边界。
- 不需要改现有打印内容结构。

缺点：
- 需要把 DOM 测量结果映射回 canvas 像素坐标。

### 方案 B：完全靠缩放避免分页

- 导出前持续缩小整体比例，直到每题都能放下更多内容。

优点：
- 实现简单。

缺点：
- 会牺牲可读性。
- 无法从根本上避免多题场景下的中间切断。

### 方案 C：更细粒度按 G/H/I 区块分页

- 不是按整题，而是按题内安全区块分页。

优点：
- 对超长题更精细。

缺点：
- 工程量更高。
- 当前问题优先级不需要做到这么细。

## 决策

采用方案 A。

## 设计细节

### 1. 导出块标记

- 为 `PrintItems` 中的每道题外层包裹元素增加 `data-print-item` 标记。
- 移动端导出时，从隐藏的 A4 导出容器里查询这些块。

### 2. 块测量

- 使用 `getBoundingClientRect()` 读取每题相对导出容器顶部的位置与高度。
- 再按 `canvas.height / target.scrollHeight` 的比例映射到 canvas 像素空间。

### 3. 分页算法

- 维护当前页起点。
- 逐题检查：
  - 如果当前题能放进当前页，继续。
  - 如果放不进，但单题高度小于单页高度，则在这题前换页。
  - 如果单题高度本身超过单页高度，则只对这一题退回到固定页高切片。
- 最后一页收尾时补上剩余高度。

### 4. 余白策略

- 允许页尾在题块边界留下适度空白。
- 不主动为“填满页面”去切断下一题。
- 这比当前“看起来紧凑但内容被截掉”更符合打印可用性。

### 5. 范围控制

- 本次只优化移动端 PDF 分页。
- 不改桌面端 `window.print()`。
- 不引入新的 PDF 渲染库，也不改打印页内容结构。

## 验证

- 复测此前被切断的题，普通题应整题落在同一页。
- 同一批题的页数可能比上一版略增，但不应再出现题内硬切。
- 对极端超长题，允许仍然分页，但应是必要的兜底行为。
